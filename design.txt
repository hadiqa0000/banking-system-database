Table: Bank
1. Database Constraints

bank_id:	Primary key, auto-generated identity, NOT NULL
legal_name:	NOT NULL, UNIQUE
bic:	NOT NULL, UNIQUE, length must be 8 or 11 characters
routing_no:	Nullable, UNIQUE if present, must be exactly 9 characters
country_code:	NOT NULL, must reference an existing Country
created_at:	NOT NULL, defaults to CURRENT_TIMESTAMP
bank_status:	NOT NULL, must reference an existing BankStatus
headquarters_city:	NOT NULL
headquarters_address:	NOT NULL
license_number:	NOT NULL, UNIQUE



2. Business Rules

These describe how banks operate in the real world.

Identity
Every bank has exactly one legal name.
A legal name identifies one banking institution.
Every bank has one primary headquarters.
A bank belongs to exactly one home country.
Every bank is issued exactly one banking license by its home country's regulatory authority.


SWIFT/BIC
Every bank participating in international transfers should have a valid SWIFT/BIC code.
The country portion of the BIC must match the bank's country.
Branch codes (last three characters) are optional.
Multiple banks cannot share the same BIC.


Routing Number
Routing numbers are only applicable in banking systems that use them.
Banks in the United States should have a routing number.
Banks outside countries that use routing numbers normally should not have one.
Routing numbers identify payment routing rather than the bank itself.



Headquarters
Headquarters city must exist within the selected country.
Headquarters address should correspond to the headquarters city.
Banks have only one registered headquarters in this model.


License
A bank cannot legally operate without a banking license.
License numbers are unique within the issuing authority.
License format depends on the country's banking regulator.


Status
Banks normally begin as Active after licensing.
Suspended banks remain in the database but have restricted operations.
Closed banks remain for historical records.
Status changes over time but only one current status is stored.


Creation Date
The creation date represents when the bank entity was established.
Banks cannot have future creation dates.
Older banks are generally more common than newly established ones.

BIC/SWIFT Code


bic country code must equal country_code.
Generate valid SWIFT codes.
Structure:

CC = ISO country code
LL = location code

Generation rules:

Country code must match country_code.
Use 8-character codes for headquarters when possible.
Use 11-character codes for branches occasionally (e.g., 20–30% of records).
Never generate duplicate BICs.

Creation Date

Generate establishment dates between approximately:

Use a weighted distribution so that:

Very old banks (before 1950) are uncommon but present.
Most banks are established between 1960 and 2015.

Bank Status

Use a weighted distribution rather than uniform randomness.
 correlate status with age:

Older banks have a slightly higher chance of being Closed due to mergers, acquisitions, or liquidation.
All unique fields (legal_name, bic, routing_no, license_number) must remain globally unique across the dataset.




Table: Branch
1. Database Constraints
Column	Database Constraints
bank_id	NOT NULL, must reference an existing Bank, ON DELETE CASCADE
branch_id	Auto-generated identity, part of composite primary key
name	NOT NULL, unique within the same bank (bank_id, name)
region	NOT NULL
branch_address	NOT NULL
country_code	NOT NULL, must reference an existing Country
Primary Key	(bank_id, branch_id)
Unique Constraint	(bank_id, name)
2. Business Rules

These describe how bank branches operate in reality.

Ownership
Every branch belongs to exactly one bank.
A branch cannot exist without its parent bank.
If a bank ceases to exist, all of its branches cease to exist (represented by ON DELETE CASCADE).
Geography
A branch normally operates in the same country as its parent bank.
Some multinational banks may operate branches in foreign countries.
Every branch belongs to one country.
Every branch has one physical address.
The branch's address must exist within its assigned country.
The branch's region should correspond to an administrative region, province, or state within its country.
Branch Name
Branch names only need to be unique within the same bank.
Different banks may have branches with identical names.
Branch names are often based on their location rather than being globally unique.

Examples:

Downtown Branch
Central Branch
Airport Branch
Financial District Branch
North Branch
Branch Operations
A bank must exist before any branches can be created.
Every operational branch represents a physical banking location.
A bank may have zero, one, or many branches.
Larger banks generally operate more branches than smaller banks.

Parent Bank Selection

Each generated branch must reference an existing bank_id.

Generation logic:

For each bank:
    Generate N branches

Never generate orphan branches.

Country Selection

By default:

branch.country_code = bank.country_code

Recommended distribution:

95% of branches are in the same country as their parent bank.
5% belong to multinational banks and may exist in another country (only if your simulation models international operations).

If your project does not model multinational banking, enforce:

branch.country_code == bank.country_code

Table: Role
1. Database Constraints
Column	Database Constraints
bank_id	NOT NULL, must reference an existing Bank, ON DELETE CASCADE
role_id	Auto-generated identity, part of the composite primary key
role_name	NOT NULL, unique within the same bank (bank_id, role_name)
Primary Key	(bank_id, role_id)
Unique Constraint	(bank_id, role_name)
2. Business Rules

These describe how employee roles exist within a bank.

Ownership
Every role belongs to exactly one bank.
A role cannot exist without its parent bank.
If a bank is deleted, all associated roles are deleted.
Role Definitions
A role defines a job position within the bank's organizational structure.
Multiple employees may hold the same role.
A role may temporarily have no employees assigned.
Role names only need to be unique within the same bank.
Different banks may have roles with identical names.
Organizational Structure
Every bank should have a set of core operational roles.
Larger banks typically have more specialized roles than smaller banks.
Executive roles are relatively few compared to operational roles.
Specialized roles (e.g., Investment Analyst) only exist in banks that offer those services.
Role Lifecycle
Roles are usually created after the bank is established.
Roles remain even if no employee is currently assigned.
Banks may add or remove roles as their organizational structure changes.

Cross-Column Consistency Rules

Ensure generated records satisfy the following:

bank_id must reference an existing bank.
The parent bank must exist before any roles are generated.
role_name must be unique within each bank.
Different banks may reuse the same role names.
Every bank should include at least the core operational roles.
Specialized and executive roles should become more common as the bank's organizational complexity increases.
Dataset-Level Generation Rules

To improve realism:

Every bank should have a consistent organizational hierarchy, with executive, managerial, operational, and support roles.
Large international banks should have a broader variety of specialized roles than local community banks.
Avoid unrealistic combinations, such as a bank having only executive roles or only IT roles.
Since employees will reference this table, generate all roles before generating employee records to ensure valid role assignments.
